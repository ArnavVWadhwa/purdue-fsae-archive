  • General feedback
    • If aiming at more final design type of presentation → polish up details, such as external references
    • Hand-draw sketches → on paper and then scan, looks better
    • If aiming for design judge audience → more about why decisions were made → if doing for some do for all decisions
    • Make sure showing enough in presentation that judge will understand but not too much that judge will be overwhelmed/bored
    • Take presentation slower, be more confident
  • Future
    • If serious about electronic integration, will need to get more people with relevant knowledge
    • Will probably want a standardized communication bus/power supply system for integrated electronics
  • Harness
    • Michael Tam asking question
      • How to make sure everything in excel and rapid harness is matching 1-1
        • Hunter answer → excel sheet good but thinking abt migrating fully to 1 software
      • Maybe dump data from rapid harness into csv and match with excel sheet
      • Big thing about making process is air tight
    • Wire sizing
      • Maybe not important enough for presentation in its entirety
      • Big points only
      • Table can be in appendix along with rapid harness diagrams
      • Make sure getting correct amount of current at higher temps
      • Look out for anything funky happening at higher temps
        • Is there any danger
      • Temp could be a little concerning for wires but not sure
    • Module locations
      • Make sure harness is waterproofed
    • Power accounting
      • Stator output → dont say not sure, say that datasheet doesn't provide specifics
    • Pneumatics
      • what is it for why is it there
    • Have to map out timings on off duty cycle periods for valve
      • Component to smooth out signal
  • Dash
    • Backing good → reusable connectors, make sure components sealed and secured, easier to waterproof and secure than sealing each component individually
    • GPS position not bad
    • Clutch paddle 
      • Feedback good
    • If data is not available/hard to get (e.g. cylinder temp), find something else to measure that we can derive our wanted data from
  • CAN Expander
    • Include in appendix as much as possible
    • Talk about how data connections happen
    • How fits into overall harness architecture
    • Spend decent amount of time on each component of the expander
    • Have software mapped out
    • documentation might be needed for firmware (clock, data rates, etc)
    • have important/sensitive data on the ECU
      • utilize ECU's priority settings
      • CAN buses have priority levels as well, maybe group signals/data into priority groups. (e.g. criticial control signals go to bus 0, experimental/test data go to bus 1)
        • this relieves bus of traffic and de-clutters system
  • DAQ (basically all CAN Expander feedback applies here as well)
    • read up on DSP/Signal Processing before tackling DAQ head-on, this will be a complex project
    • DAQ system = senior level project → fuels my god complex (frfr)
    • Voltage swing
      • Suspension quesiton → is swing in mV enough for suspension system
    • Antenna position is complicated
      • Ideally not surrounded by metal
    • PCB signal integrity
      • Spacing of components/traces is big
      • Just follow good routing practices (keep lines short, space traces out, etc.)
      • component orientation, packaging
    • Hardware vs Software filtering (ok this 
      • Balance response time
      • If hardware → microcontroller doesn't have to do it, saving processing power (may not be a big save tho)
      • Software filter: slower data acquisition but flexible → better option because filtering after collecting data not real time
  • Dyno
    • Elaborate on NI USB-6210
      • What is it
      • Some context
  • Runsheet
    • Show some of the software in the presentation
      • Software architecture
    • Spelling error → "humidity"
