Automated load control will help our team achieve its goals by providing a system that speeds up our teams ability to accomplish dyno tasks. Because many tasks on the dyno involve tuning the engine over both the throttle and RPM range it is necessary to have a system that can accurately and quickly reach these targets to minimize error and time taken to complete. The goal of the automated load control project will be to allow the user to set an RPM target then use a system that will automatically adjust the water load to the DynoMite water brake to hold the engine at that RPM. This is a critical component for the dyno tuning, because often the RPM will adjust based on throttle and fuel inputs, so having a system that can adjust the load dynamically will ensure the testing and tuning are accurate.

Automated Load Control Requirements:

  1. Take user inputs from M1 Tune
  2. Control the flow of water to the water brake

At the start of the 2020 Fall semester we had a run down dyno cart with an engine that didn't run, an oversized water pump, no water control valve, and an un-calibrated load cell.

Research:

For the water pump Dominic did the flow calculations to correctly size the pump.

For the flow control there are two main components:

  1. Valve
    1. Ball Valve - This valve only turns 90 degrees which means that a servo motor could control this type of valve. Because of the exponential nature of the flow for a ball valve it is not viable for the dyno.
    2. Gate Valve - This valve has up to 5 rotations between full open and full close. This allows for a linear application of flow to the water break. Meaning that the load can applied with finer turns to the handle.
  2. Flow Control
    1. Servo Motor - Only capable of about 180 degrees of rotation. Low torque. Not capable of moving through the entire range of a gate valve. We do have a Savox SS2291SG
    2. Stepper Motor - Capable of achieving 360 continuous motion and able to apply 3Nm of torque. https://www.amazon.com/gp/product/B00PNEPW4C/ref=ppx_yo_dt_b_asin_title_o03_s00?ie=UTF8&psc=1

For the Load Measurement we use a 100lb load cell connected to a IZZE Load Cell Amplifier which communicates the load cell values over CAN.

This function reads the CAN message from the IZZE amp.

I did have difficulties sending a configuration to the the IZZE, so in order to read the values correctly from the load cell you need to scale the values by 0.009 in I2.

Something that should be researched is how to communicate with the IZZE because this scale could be sent over CAN on startup, but any attempt to use the M1 system or an AD2 to communicate over CAN did not succeed.

With the selection made for a gate valve and a stepper motor, Carson Williams designed a 3d printable mount for the gate valve and stepper motor assembly. The files can be found in the mefsae drive 2021 powertrain folder.

Design Changes:

From our results, we decided to revert back to all of the equipment that was used to run the dyno when it was working. We re-installed the massive water pump and the old flow control valve.

This is our final design for the stepper motor and value. The stepper sits on four standoffs that set the position of the stepper shaft in the right spot for the set screw to clamp the axle down. We marked the opening of the valve with the yellow paint pen. That should be the starting position.

It's important to note that every time the dyno is turned off you must reset the position to 0. The easiest way to reset the position to 0 is to set Percent Load to 0, then unplug the stepper motor and manually set the position so the yellow stripes line up.

Once we made those changes the dyno seemed very responsive and worked like we expected it to.

Then I wrote a PI controller for the stepper motor:

A Proportional Integral (PI) Controller was used first because Proportional controllers are a common control method that fit well into our application. The goal of a proportional controller is to measure the Error (Actual Position - Target Position) and use that error to add or subtract from the actual position of the stepper based on the target. In this situation we also need a component that will help us reduce the error as fast as possible otherwise the RPM could climb very quickly before the controller/stepper had time to react. For this we added the integral part to the controller. A PI controller should allow the stepper to calculate its target and adjust the position very quickly as to not let it climb.

This feature is out of date as of 12/5/2020 because it has been updated for the PID controller

Also error in coding: The Error time was continuously growing as time went on, but we were also growing our Integral value over the iterations of Aim Update meaning that the "Integral value" was exponentially growing over time which should not be the case. The fix was to remove the Error Time variable and replace it with 0.02 which is the amount of time in seconds that the Aim Update function is cycling. This value will have to be updated if the function update frequency is changed.

Results:

5Hz Aim Update Rate

After this test we thought that the Aim Update function was not being allowed to update fast enough. We thought that this is what was causing the overshoot and oscillation. So we updated the scheduled frequency from 5Hz to 50Hz.

In this test we also confirmed that the system does respond to changes in throttle and stays near the RPM Target of 6000RPM.

50Hz Aim Update Rate

In this result we can see that the position of the stepper motor moves from 0 (fully closed = no load) to between 35-45 (about 45%-55% open) this causes the RPM to stutter between about 5500 - 6500 RPM. Our target in this run was 6200RPM. In order to improve the controller, we need a damping coefficient that would force the stepper motor to not back all the way down to 0 or climb all the way back to a peak. This means that we should introduce the derivative component to make it a PID controller. The derivative component takes the slope of approach and tries to adjust the aim to slow down the approach. This should help our situation because we are currently overshooting and oscillating around our desired set point.

Also another error in the coding was using the PID Aim as an update feature to the position (Aim = Convert.ToInteger(Position + PID Aim)). This actually converted the aim controller to an I controller not a PI controller which caused oscillations and added to the issue when Ki was added. This was updated to Aim = Convert.ToInteger(PID Aim) so the PID aim becomes the actual output making it a true PID controller.

Finally I updated the controller to be a TRUE PID loop
