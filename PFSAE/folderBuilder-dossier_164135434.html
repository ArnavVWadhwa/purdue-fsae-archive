Purdue Formula SAE : folderBuilder dossier

Created by Unknown User (agarw162), last modified by Unknown User (rkumaran) on Dec 03, 2020

folderBuilder.py is the perfect example of what python is good for. A quick and dirty script doing something not particularly advanced. folderBulder.py essentially parses everything in linuxConfig into system environment variables in a bash script, then submits the bash script to the SLURM scheduler. 


There are a few flags linuxConfig HAS to have, otherwise folderBuilder.py will crash. These flags define things you need for every simulation run, so if you don’t have these flags, why are you using folderBuilder in the first place? folderBuilder.py doesn’t care what the value of these flags are set to, just that they exist. This is largely because folderBulder.py makes no attempt to verify the validity of your flag inputs. Preferably, don't set these flag values to anything that could be interpreted as None (note, Python will never treat 0 as None, like some other lanugages may treat 0 as null).

 

CLUSTER – this defines the cluster you’re planning on using. “LOCAL” is a valid option, and then folderBuilder.py will skip all the additional logic checks it needs for a cluster run.  

 

STARLOC – where you STAR installation is.  

 

LICPATH – where your license is.  

 

CP – Class path for the macros.  

 

Syntax for folderBuilder.py 

 

folderBuilder.py <job1 job2 job3 …...> where the additional job numbers are dependencies. New jobs will not begin until the dependencies are complete. This is a great way to queue up a bunch of sims, without having to wait for one simulation to finish, before submitting the next.
