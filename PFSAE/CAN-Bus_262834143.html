Notes:

  • FSAE/Industry standard is yellow CAN high and green CAN low

         

  • Terminated on both ends by 120Ω resistors
  • Distance from the CAN bus to any given node must be <30 cm

CAN Frames:

CAN bus communication is done through frames, which has 8 parts.

  • SOF: "Start of Frame -  "dominant 0", which tells the CAN bus that a node is about to transmit data (1 bit)
  • ID: Frame identifier - lower value = higher priority (11 bits for CAN2.0A, 29 bits for CAN2.0B)
  • RTR: Remote Transmission Request - shows whether node is sending data or requesting data from another node (1 bit)
  • Control: Contains 2 parts:
    • Identifier Extension Bit (IDE): "dominant-0 for CAN2.0A (1 bit)
    • Data Length Code (DLC): Specifies Length of data being transmitted
  • Data: Data from sensors and nodes in the CAN bus
  • CRC: Cyclic Redundancy Check - ensures data integrity
  • ACK: Acknowledgment - indicates if node has received data correctly
  • EOF: End of Frame


Example CAN2.0B data


Raw CAN data isn't readable, so you need to decode the CAN data into actual engineering values for it to make sense.

Decoding CAN Data

The encoding process for CAN data is often proprietary, and you'll have to look through some databases to figure out how to decode the CAN data for your specific application.

Generally the CAN data will be scaled by some value, and then it will be offset by some value.



CAN Bit Timing Requirements

The various nodes of the CAN bus need to be sending data to the ECU at a rate at which it can read it.

In the CAN protocol, the bit rate, bit sample point, and number of samples per bit can all be coded into your Design Environment.
