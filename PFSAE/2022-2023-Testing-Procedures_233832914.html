Purdue Formula SAE : 2022-2023 Testing Procedures

Created by MacNab, Zachary Robert, last modified by Unknown User (amattu) on Aug 22, 2022

Purpose

Currently, I think we don't know enough about the package that we are physically sending out on track. I don't think that great documentation exists of what we set out on track, and I think we can make it a lot easier and possibly integrate it with MoTeC data so it's really seamless. For instance, after seeing some of RLL's testing procedures and reflecting, I thought that spring testing could've gone a bit better. By setting up the car properly, then we know we have a package that we designed to, and if there is anything wrong with that package then we know it and can account for it in our analysis of what the car is behaving like. 

Outline

There are two main things that I want to integrate to the team

  • Set-up Book
    • The idea behind the set-up book is that it records every single variable that we can change on the car. This includes but is not limited to:
      • Suspension:
        • Weights
        • Camber
        • Toe
        • Springs
        • Damper Settings
        • Pressures
        • ARB
        • Brake Bias?? (This is a variable to be filled out post run?)
      • Aero:
        • FW Flap Angles
        • RW Flap Angles
        • Ride Heights
      • Powertrain:
        • Differential Tuning
          • Preload
          • Ramp Angles
          • Number of clutch plates
        • Final Drive Ratio (not generally easily tunable)
        • Tune? Do we want a list of tunes? Or want to refer to some file?
      • Chassis:
        • Pedal box position
        • Throttle mounting position
    • In the set-up book, there can be other calculated sheets as well. Things like CoP, front weight dist., cross weight dist., etc. can all be calculated 
  • Run-sheet (could be integrated to the set-up book)
    • Keeps track of all changes and weather – still want to more clearly define what this holds and why we need a separate sheet

From here, I think we can correlate our ADAMS simulations better. If we correlate the sims, we can trust the sims more, use the sims to try a bunch setups and find an optimal set-up for each event in the lab rather than on valuable testing time.
