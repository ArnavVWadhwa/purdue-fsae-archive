Purdue Formula SAE : Making Large Changes to CAD Models

Created by Unknown User (hammerly), last modified on May 01, 2023

Building Robust CAD The First Time

In order to keep your model's dependencies simple, try to reference the oldest geometry possible when creating a model. I mention this in this page: 1. Creating a Wireframe but it's relevant to anyone building a model. For example, be intentional about using reference geometry (such as suspoints) instead of created geometry in a part (such as created parts of the wireframe). This is an easy mistake for someone to make, but it may cause problems downstream when the non-reference geometry parent is changed or deleted.Â 

With the mouse hovering over the item you'd like to reference, right-click then click "Select from List...". This will open a window displaying all the possible items that this feature could reference. Be diligent and select reference geometry as parents of a feature!


How to Determine if Big Changes Are Necessary

Is your model still up to date? Because we practice top-down CAD modeling (learn more about what that is here https://www.3dcadworld.com/cad-basics-top-down-modeling/ or here NX CAD Practices ), any upstream changes to reference geometry will trickle down into your model, and it will fully update automatically. For example, if suspension points change slightly, the tubes and tabs should all update accordingly.

There have been instances of the associative links between parts breaking, rendering the top-down method useless. This is a huge problem if you need to make a fundamental change later on. Ideally, all the features in your part should show a green check mark in the Up to Date column in the part navigator.

Dependency Browser

Before diving into this, it's important to understand the dependencies within a model. There's a couple ways to do this. The easiest way is to click the feature within your part tree. Once highlighted, any features which the selected one depends on will be shown with purple text. Any features which depend on the selected one will be shown with blue text.

Another way to visualize these dependencies is by using the Browser tool. In NX 1946, Menu>Information>Browser pulls this up. This will initially provide a blank-looking window, until a feature in the part tree is selected. From there, the Browser tool shows any immediate parents of the selected feature. This tool allows you to get a full view of the feature's history by clicking the next to any feature, displaying that feature's parents.

Replacing Items

If I want to replace Datum Plane (230) which drives the Trim Body (231) and (233) features, there's a few things that need to be done.

  1. Create the replacement geometry. In this case, I've created a Fixed Datum Plane that will replace Datum Plane (230). Place this feature above the feature you wish to replace in the part tree.
  2. Make the feature you wish to replace the "Current Feature." This will prevent any downstream features from updating.
  3. Replace the feature.
  4. Check that the feature has been replaced properly by checking that the dependencies have been transferred to the new feature.

Updating Intentionally

Once you've replaced a feature, it's important to be very diligent when updating the rest of the model. There's a couple ways to go about this. By default, NX models update automatically, so that's why I prevented downstream features from updating by using "Make Current Feature" in step 2 of Replacing Items. You can step through the rest of the model with Make Current Feature, and update the rest of the model feature by feature.

Alternatively, another way to ensure that the model only updates when you want it to is to prevent it from updating altogether temporarily. With Tools>Delay Model Update, NX will stop updating automatically. You can make changes (such as replacing features) and either step through the model with Make Current Feature or update all of them at once. I prefer to step through the model when updating features because NX is less likely to throw a Memory Access Violation error (what happens if a request for NX requires too much computing power at once).
