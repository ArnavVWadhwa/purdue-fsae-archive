Purdue Formula SAE : Pre-requisites

Created by Unknown User (agarw162), last modified by Zhang, Chenkai on May 07, 2023

There are a few fundamental skills you’re expected to have before jumping into this documentation. This document should tell you what you need, and where you can get the resources to learn those skills.  

 

STAR-CCM+ 

 A lot of this documentation builds on the assumption that you know how to start STAR, and set up simple simulations. This is going to be more geared towards explaining our existing simulation, automation, and development methods, and isn’t going to try to teach you how to code in Java or how to build a CFD simulation from scratch. There are plenty of better resources out there that can do that. If you need more practice with STAR, head over to the steve portal, or the support center (or whatever Siemens is calling it now).  

 

We (I) have always gravitated towards staying on the latest version of STAR-CCM+. I would strongly recommend continuing with this practice. It’s a little annoying when people don’t want to update STAR, but making sure you’re not deviating away from the latest release of the API, and the package itself makes sure you’re not going to be forced into re-writing large chunks of code or reworking the STAR environment. Re-writes to accommodate API changes is something I’ve had to do, and reworking the STAR environment is something that’s a real possibility, considering how Siemens generally doesn’t shy away from deprecating major parts of the workflow if they don’t have faith in it (regions based meshing).  

 

Pick a single software platform to commit to (Windows vs Linux), and spend all your resources developing to that platform. Probably want to do Linux, since that’s what RCAC nodes are built on. There’s very little benefit in my experience to targeting multiple platforms, and ads a lot of additional potential for debugging headaches. I’m going to assume you’re using a Linux platform.   

 

Java 

 It’s important to have a basic familiarity with Java before jumping into our CFD methods. You don’t need to be a Java god, and as of right now our codebase rarely uses any of the languages more esoteric features. A crude understanding of how object-based programming works, and how Java handles data structures like arrays, lists, and collections should be all you need to know to understand the code. There are plenty of guides and tutorials online that should get you up to speed. If you’ve taken CS 159, and spent 15 minutes reading about OOP online, you should be good to go to handle anything our codebase can throw at you.  

 

STAR’s Java API is very poorly documented in my opinion. The documentation is more a list of methods and functions with no additional descriptions, than actual documentation. Thankfully STAR’s method names are somewhat self-explanatory, so be prepared to have to figure out what a particular method does or doesn’t do by yourself.  

 

It’s easy to keep on the latest release of the JDK/JRE, or to jump on JetBrain’s latest JBR release. STAR installs all ship with a JDK, just use that. You can find it in STAR’s program files. I’ve had issues with attaching Java debuggers to STAR if you’re using a JDK/JRE that doesn’t match the version that ships with STAR. You can find the JDK that ships with STAR.  

 

IntelliJ is a solid IDE for Java. I’d strongly recommend using it. Be sure to know how to link IntelliJ’s debugger with STAR (again, Siemens support center) 


Python 

With the way our python scripts are currently set-up, you don’t even really need to know python to be able to work with the scripts. We’re using python for the extremely simple and straightforward aspects of our code. If you figure out the Java, you’ll have no issues with the python.  


Learn python the hard way is a fantastic book to use to learn python though. It’s not a long process, and it’s a very inviting language to learn, but it does feel a lot different from C syntax languages like C/C++/Java.
